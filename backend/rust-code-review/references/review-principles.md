# Review Principles

- Review for correctness first.
- Review design before style.
- Prefer small, targeted changes.
- Understand every changed human-written line unless scope is explicitly narrowed.
- Say when generated files, data files, or large files are skipped.
- Tests are code and must be reviewed.
- Review concurrency and async behavior for Rust backend changes.
- Comments should explain why, not restate what code does.
- Ask for clarification when code cannot be understood safely.
- Compliment good changes when it helps reinforce the right pattern.
- Do not accept code that degrades code health.
